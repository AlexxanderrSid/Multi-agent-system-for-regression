# Описание и обоснование методов

## 1. Постановка задачи

Задача — **регрессия на табличных данных**: предсказание числового целевого значения (`target`) по набору признаков. Метрика качества — **MSE (Mean Squared Error)**, чем меньше — тем лучше. Данные представляют собой Airbnb-подобный датасет с числовыми и категориальными признаками, датами и пропущенными значениями.

---

## 2. Выбор архитектурного паттерна: MLE-STAR

Система построена по паттерну **MLE-STAR (Machine Learning Engineer — Search & Targeted Refinement)**. Ключевая идея: вместо одного монолитного агента задача решается итеративным циклом специализированных агентов, каждый из которых отвечает за строго определённый шаг пайплайна.

**Обоснование выбора MLE-STAR:**

- Разделение ответственности позволяет каждому агенту иметь узкоспециализированный system prompt, что значительно повышает качество генерируемого кода по сравнению с универсальным промптом.
- Outer/inner loop с ablation study даёт систематическую обратную связь о вкладе каждого компонента, направляя рефайнмент туда, где он даёт наибольший прирост.
- Автоматический дебаг через `DebuggerAgent` делает систему устойчивой к ошибкам выполнения сгенерированного кода — неизбежным при работе с LLM.

---

## 3. Выбор LLM: Qwen2.5-Coder-7B-Instruct

### Модель

**Qwen2.5-Coder-7B-Instruct** (Alibaba Cloud, лицензия Apache 2.0), квантизованный формат **GGUF Q4_K_M** (~4.7 GB), запускается через `llama-cpp-python` с CUDA-ускорением.

### Обоснование

**1. Специализация на коде.**  
Qwen2.5-Coder — это серия, дообученная специально на задачах генерации, отладки и объяснения кода. В отличие от общих языковых моделей, Coder-вариант значительно точнее генерирует синтаксически и семантически корректный Python-код, что критически важно для системы, где код исполняется автоматически.

**2. Оптимальное соотношение размер/качество.**  
Размер 7B параметров — компромисс между вычислительными требованиями и качеством. Модель умещается в VRAM GPU T4 (16 GB) в формате Q4_K_M без потери существенного качества. Модели меньшего размера (1B–3B) дают заметно более слабый код; модели большего размера (13B+) не помещаются в стандартную среду Colab.

**3. Open-source и Apache 2.0.**  
Лицензия Apache 2.0 разрешает коммерческое использование без ограничений. Это принципиальное отличие от проприетарных API (OpenAI, Anthropic), которые требуют передачи данных на внешние серверы, что неприемлемо в ряде корпоративных сценариев.

**4. Локальный инференс через llama-cpp-python.**  
`llama-cpp-python` обеспечивает эффективный инференс GGUF-моделей с поддержкой CUDA. Это устраняет задержки сетевых вызовов, делает систему воспроизводимой (нет зависимости от доступности API) и позволяет контролировать версию модели.

**5. Chat Completion API.**  
Модель поддерживает стандартный интерфейс `system / user` ролей, что позволяет передавать строгие инструкции в system prompt (например, "Output ONLY Python code") отдельно от контекста задачи в user prompt. Это улучшает следование формату по сравнению с prompt-completion форматом.

### Параметры инференса

| Параметр | Значение | Обоснование |
|---|---|---|
| `n_gpu_layers=-1` | все слои на GPU | Максимальная скорость на T4 |
| `n_ctx=4096` | размер контекста | Достаточен для кода + промпта |
| `n_batch=512` | batch size | Оптимум для T4 |
| `temperature=0.2` | низкая температура | Детерминированный, точный код |
| `max_tokens=900–1300` | зависит от агента | Баланс полноты и скорости |

## 4. Агенты и обоснование их ролей

### 4.1 DataAnalystAgent

Отвечает за очистку данных: удаление дубликатов и нерелевантных столбцов, заполнение пропусков, конвертацию дат. Вынесен в отдельный агент, так как очистка требует специфических знаний о правилах обработки данных без утечки (`fit` только на train) и должна выполняться до feature engineering.

### 4.2 FeatureEngineerAgent

Генерирует новые признаки на основе RAG-контекста и plan trajectory. Plan trajectory — список уже опробованных подходов — предотвращает повторение одних и тех же идей в разных итерациях. Агент также реализует роль Planner в inner loop: предлагает новый план feature engineering после каждой неудачной попытки.

### 4.3 ModelAgent

Обучает LightGBM и XGBoost. Выбор этих библиотек обусловлен их доминированием на соревнованиях по табличным данным: они нативно обрабатывают пропуски, устойчивы к выбросам и требуют минимальной настройки для получения хорошего baseline. Агент реализует два режима: начальное обучение (`run`) и рефайнмент по конкретному плану (`run_refined`).

### 4.4 AblationAgent

Ключевой элемент MLE-STAR. Генерирует код, который тестирует три вариации пайплайна с отключёнными компонентами, определяет наиболее значимый из них и формирует план следующего улучшения. Ablation memory предотвращает повторное изучение одних и тех же компонентов.

### 4.5 EnsembleAgent

Итеративно предлагает и реализует стратегии ансамблирования (простое усреднение, взвешенное по 1/MSE, стекинг с Ridge, rank averaging). История попыток позволяет каждый раунд предлагать новую стратегию вместо повторения уже опробованных.

### 4.6 DebuggerAgent

LLM-генерируемый код содержит ошибки в ~30–50% случаев при первой попытке. DebuggerAgent получает код и traceback и генерирует исправленную версию. Три попытки (`MAX_DEBUG=3`) охватывают большинство простых и умеренных ошибок; при исчерпании попыток система продолжает работу без этого шага.

## 5. RAG (Retrieval-Augmented Generation)

Система использует простой keyword-based retrieval по четырём документам базы знаний:

- **preprocessing** — правила очистки данных без утечки
- **feature_engineering** — рекомендованные признаки для Airbnb-подобных датасетов
- **models** — рейтинг моделей для табличной регрессии
- **ensemble** — стратегии ансамблирования

**Обоснование keyword-based подхода:** для небольшой базы (4 документа) и узкоспециализированных запросов (каждый агент запрашивает строго по своей теме) простой keyword-поиск даёт результаты, сопоставимые с dense retrieval, при нулевых вычислительных затратах и без зависимостей (нет нужды в векторной БД или embedding-модели).

## 6. Безопасное выполнение кода

Генерируемый LLM код выполняется через `exec()`. Для минимизации рисков реализованы:

- **Статический анализ перед выполнением** (`check_code_safety`): паттерны для запрещённых вызовов (`os.system`, `subprocess`, файловая запись).
- **Предупреждения об утечке данных** (`check_data_leakage`): паттерны `fit_transform` на тестовых данных.
- **Изолированный контекст**: `exec` выполняется в локальном словаре, обновляющем общий контекст системы только при успехе.
- **Захват stdout**: через `redirect_stdout` перехватывается вывод для извлечения метрики.

---

## 7. Метрика и мониторинг

**MSE (Mean Squared Error)** выбрана как метрика задачи регрессии. В отличие от MAE, MSE штрафует за крупные ошибки квадратично, что соответствует типичным требованиям в задачах ценообразования.

`Monitor` логирует каждое событие системы с временной меткой, отслеживает лучший результат и по итогу формирует отчёт с полной историей метрик, количеством LLM-вызовов и временем выполнения.

---

## 8. Бенчмарк

Перед запуском мультиагентной системы выполняется 5-fold cross-validation четырёх baseline-моделей (Ridge, RandomForest, GradientBoosting, LightGBM с дефолтными параметрами). Это позволяет объективно оценить, насколько мультиагентный подход превосходит наивные baseline, а не только сравнивать агентов между собой.
